# 自动化状态持久化计划（2026-03-21）

## 目标
- 为现有自动化系统增加“重启可恢复”的轻量持久化能力，避免关键时间节点仅保存在内存里。
- 优先解决“命令时间/冷却/待处理标记”在进程重启后丢失的问题。
- 严格遵循最小修改原则：不重做调度器，不引入复杂外部依赖。

## 当前现状
- `Scheduler` 仅维护内存任务，重启后所有已安排任务都会丢失。
- 多个插件依赖内存状态记录冷却、今日完成情况、待结算标记、下一次尝试时间。
- `app.py` 是统一创建插件与注入依赖的入口，适合接入统一状态存储。

## 已识别的受影响模块
- `xinggong.py`：启阵、助阵冷却、深度闭关联动、观星劫持窗口。
- `yuanying.py`：探寻裂缝冷却、元婴出窍/待归窍结算状态。
- `chuangta.py`：今日是否已闯塔、待补做标记、元婴联动状态。
- `zongmen.py`：每日点卯、传功次数、是否已禁用。
- `garden.py`：缺种/无空地等轻量状态。
- `biguan.py`：主要依赖延迟任务，重启后“下一次闭关时间”会丢失。

## 初步方案
- 新增统一状态存储层，例如 `xiuxian_bot/core/state_store.py`。
- 采用单个本地 JSON 文件保存各插件命名空间状态。
- 每个插件仅持久化“关键恢复字段”，由插件自己负责 dump/restore。
- 启动时加载状态，再由各插件基于已恢复字段重新安排内存任务。
- 目标是“尽量恢复正确行为”，不追求对所有历史消息 id 的 100% 精确复原。

## 需要你确认
1. 持久化范围
   - a. 先覆盖关键插件：`xinggong / yuanying / chuangta / zongmen / biguan`（推荐）
   - b. 所有插件都做
   - c. 只做你最关心的几个插件

2. 持久化介质
   - a. 本地单个 JSON 文件（推荐，改动最小）
   - b. sqlite

3. 恢复精度
   - a. 持久化“时间点 + 今日计数/待处理标记”，启动后尽量恢复（推荐）
   - b. 只持久化时间点，不恢复计数和待处理标记

4. `biguan` 是否纳入第一批
   - a. 纳入，重启后也恢复下一次自动闭关（推荐）
   - b. 暂不纳入，只做其余插件

## 已确认方案
- 持久化范围：所有插件（其中 `daily` 当前无运行态，保持无状态）
- 持久化介质：sqlite
- 恢复精度：恢复关键时间点 + 今日计数/待处理标记
- `biguan`：纳入第一批
